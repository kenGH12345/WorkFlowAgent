## Rules
<!-- PURPOSE: Prescriptive constraints that MUST be followed. Written as imperatives ("Always X", "Never Y"). Each rule should be independently verifiable. Rules are the highest-authority content in a skill — they override best practices when in conflict. -->

1. **Prefer `const` constructors everywhere** — Mark widgets and objects as `const` whenever possible. This enables Flutter's widget identity check to skip rebuilds entirely, giving free performance gains.

2. **Never call `setState()` inside `initState()` or `dispose()`** — These lifecycle methods must not trigger rebuilds. Schedule post-frame callbacks via `WidgetsBinding.instance.addPostFrameCallback` if you need state changes after init.

3. **Always specify explicit types for public API** — Use `final String name` not `final name`. Dart's type inference is excellent for local variables but public surfaces must be self-documenting.

4. **Use `key` parameters on list items** — Every widget in a `ListView.builder` or similar must have a stable, unique `Key`. Without it, Flutter may incorrectly reuse state across items during reordering.

5. **Isolate expensive computations with `compute()`** — Any CPU-bound work (JSON parsing, image processing) exceeding 16ms must run on a separate Isolate via `compute()` to prevent UI jank.

6. **MANDATORY: Standard Flutter directory structure** — A Flutter project MUST have the following top-level directories: `android/`, `ios/`, `lib/`, `test/`. These are generated by `flutter create` and are required for platform builds. Never omit or rename them.

7. **MANDATORY: Client/Server separation** — When a Flutter project includes server-side business logic (Java, Go, Python, etc.), the server code MUST be placed in a separate subdirectory (e.g. `server/`, `server-java/`, `server-go/`) at the project root. The Flutter client files (`android/`, `ios/`, `lib/`, `pubspec.yaml`) MUST remain at the project root level. The server is a **sibling** directory, NOT a parent. Never place Flutter client files inside a `client/` subdirectory — Flutter CLI tools expect them at root.

```

├── android/              ← Android platform (flutter create)
├── ios/                  ← iOS platform (flutter create)
├── lib/                  ← Dart source code

│   ├── app.dart          ← MaterialApp / root widget
```

## SOP (Standard Operating Procedure)
<!-- PURPOSE: Step-by-step workflow for the skill's domain. Numbered phases with clear entry/exit criteria. An agent following this SOP should produce consistent, high-quality output regardless of the specific project. -->

1. **Widget Creation Flow**: Identify scope → Choose StatelessWidget if no mutable state → Use ConsumerWidget (Riverpod) or BlocBuilder (Bloc) for reactive state → Extract sub-widgets when `build()` exceeds 80 lines.
2. **State Management Selection**: Local ephemeral state → `setState()` / `ValueNotifier`; Shared app state → Riverpod `Provider` or Bloc; Server-cache state → `AsyncValue` + repository pattern.
3. **Navigation**: Use declarative routing (`GoRouter`) for all navigation. Avoid imperative `Navigator.push` chains — they break deep linking and web URL sync.

## Best Practices
<!-- PURPOSE: Recommended patterns that SHOULD be followed. Unlike Rules (which are mandatory), Best Practices are advisory — they can be overridden with justification. Each entry explains WHAT to do and WHY it helps. -->

1. **Widget decomposition over deep nesting** — Break widgets into small, focused components. A `build()` method deeper than 5 levels of nesting is a code smell. Extract named widgets with their own class.

2. **Use `RepaintBoundary` for animated regions** — Wrap frequently-repainted widgets (animations, scrolling content) in `RepaintBoundary` to isolate their paint region from the rest of the tree.

3. **Prefer `SliverList` over `Column` in scrollables** — When building scrollable lists, use Sliver-based widgets. Column-in-SingleChildScrollView forces layout of ALL children at once, killing performance on long lists.

4. **Image caching with `cached_network_image`** — Never load network images without caching. Use `CachedNetworkImage` with explicit `memCacheWidth`/`memCacheHeight` to avoid holding full-resolution bitmaps in memory.

5. **Golden tests for critical UI** — Use `matchesGoldenFile()` for any screen that must not regress visually. Run golden tests in CI with consistent host platform to avoid pixel-diff false positives.

## Anti-Patterns
<!-- PURPOSE: Common MISTAKES to avoid. Each entry describes: (1) the wrong approach, (2) why it's wrong, (3) the correct alternative. -->

1. **God Widget** — Putting all logic in a single widget with 500+ line `build()`. Instead: decompose into small focused widgets, each with a single responsibility.

2. **Using `GlobalKey` for state access** — Reaching into another widget's state via GlobalKey breaks encapsulation and causes subtle bugs during tree restructuring. Instead: lift state up or use a state management solution.

3. **FutureBuilder in `build()` without caching the Future** — Creating a new Future inside `build()` triggers infinite rebuild loops. Instead: create the Future in `initState()` and store it in a field.

4. **Blocking the UI thread with synchronous I/O** — Calling `File.readAsStringSync()` or heavy JSON parsing on the main isolate. Instead: use `compute()` or `Isolate.run()`.

5. **Over-using `setState()` at the root** — Calling `setState()` high in the widget tree forces a full subtree rebuild. Instead: push state down to the smallest subtree that needs it.

## Context Hints
<!-- PURPOSE: Background knowledge that helps an agent make better decisions. Not rules or practices — just useful context about libraries, team conventions, or known limitations. -->

1. **Flutter Web vs Mobile rendering** — Flutter Web uses CanvasKit (WASM) or HTML renderer. CanvasKit gives pixel-perfect fidelity but has a ~2MB download overhead. For content-heavy sites, consider HTML renderer.

2. **Dart 3.x pattern matching** — Dart 3+ supports sealed classes, pattern matching (`switch` expressions), and records. Leverage these for exhaustive state handling instead of `if-else` chains.

3. **Hot reload limitations** — Hot reload does NOT re-run `initState()`, `main()`, or static field initializers. If state seems stale after hot reload, do a hot restart instead.

4. **Platform channel gotcha** — `MethodChannel` calls are asynchronous and can silently fail if the native side throws. Always wrap platform calls in try-catch and provide fallback behavior.

5. **Riverpod 2.x `ref.watch` vs `ref.read`** — Use `ref.watch` inside `build()` for reactivity, `ref.read` inside callbacks for one-shot access. Mixing them up causes either missed updates or unnecessary rebuilds.
